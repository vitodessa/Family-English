## Девять стратегических решений сессии

### 1. Слияние треков

Платформа английского — не отдельный проект, а **сквозной контекст** для обучения Матвея. Стартует **сразу после закрытия шага 8 наставника**, без отсрочки на осень.

### 2. Гибридная конфигурация

Платформа — **основной учебный контекст**. Карта обучения (`learning_map.md`) — **страховка от пробелов** в общем IT-образовании. Большинство заданий идёт через платформу, карта подсказывает наставнику, чего не хватает.

### 3. Архитектура единого ядра

Не event-sourcing с проектированием на все модули наперёд. **Минимальное ядро, растущее постепенно** с каждым новым модулем. Метафора, найденная в сессии: ядро как **«единый мозговой центр»**, который видит каждого пользователя и семью целиком, управляет процессом обучения.

### 4. Грамматика как сквозной слой

**Стратегия B** (не отдельный модуль). Таблица `grammar_topics` в ядре с первого дня. Все события (карточки, Speaking-ошибки, чтение) тегируются грамматическими темами. Дашборд грамматики появляется естественно к 3-4 месяцу, когда накоплены данные.

### 5. Четыре приоритетные ценности проекта

Зафиксирован новый порядок:

1. **Прозрачность процесса** — независимая диагностика реального уровня каждого. Решает боль заказчика с Green Country («6 лет занимался, не понимал реального уровня»)
2. **Учебный проект Матвея** — рост через реальный продукт с реальными пользователями
3. **Усиление существующего обучения в EnglishDom** — закрепление и практика между уроками
4. **Экономия в будущем** — горизонт от 12 месяцев, через постепенное снижение зависимости от внешних подписок

### 6. Трёхуровневая видимость данных

Архитектурное решение, закладывается с первого дня:

- **Личный дашборд** — пользователь видит про себя (для самоулучшения)
- **Семейный фид** — все видят всех, только мотивирующая информация (новые слова, активность). Цифровизация уже работающего семейного чата с ежедневной отчётностью
- **Родительский дашборд** — полная картина по детям, только заказчику и маме

**Защита Матвея:** в семейном фиде не больше, чем в текущем Telegram-чате семьи. Детальная диагностика — только родительский уровень.

### 7. Один бот — два наставника (Матвей + Никита)

Технически — общая инфраструктура (код, БД, хостинг, API). Педагогически — **раздельные системные промпты** и раздельные карты обучения. **Вариант B**, не условные ветви в одном промпте.

Никита: профилирование начинать **через 2-3 недели**, не сейчас. Темп Матвея с платформой важнее, профиль Никиты требует серьёзной работы.

### 8. Цикл отчётности

- **Еженедельный отчёт от наставника** — каждое воскресенье, полный по структуре 10 пунктов
- **Правка промпта — раз в месяц планово** (защита от over-fitting на недельные колебания)
- **Внеплановая правка — только по инициативе наставника** при серьёзной причине, заявленной в отчёте

### 9. План модулей платформы

Финальная последовательность с учётом всех ценностей:

| Этап | Модуль | Ценность |
|---|---|---|
| 0+1 | Ядро + Карточки FSRS с удобным вводом из уроков EnglishDom | Немедленное закрепление слов из живых уроков |
| 2 | Speaking | Закрывает разговор + поддерживает Матвея летом без EnglishDom |
| 3 | Writing | Та же архитектура, что Speaking |
| 4 | Дашборд CEFR | Первая видимая реализация «прозрачности» |
| 5 | Чтение | Расширяет вход контента |
| 6 | Семейный слой полный | Расширение базовой видимости |
| 7 | Аудирование | Дополняет |
| 8 | Видео через Language Reactor | Главный канал погружения |
| 9 | Произношение | Когда подоспеет технология |

## Технологический стек платформы

- **Бэкенд:** FastAPI (Матвей это знает по наставнику)
- **Фронтенд:** серверный рендеринг через Jinja2-шаблоны. Не SPA. Не React. Минималистичный UX
- **БД:** SQLite на старте → PostgreSQL по необходимости
- **Интерфейс:** веб-приложение с авторизацией и личными кабинетами на 4 пользователя
- **Алгоритм повторения:** готовый Python-пакет `fsrs` (не пишется с нуля)
- **AI:** Claude API, prompt caching
- **Голос (когда дойдём до Speaking):** Whisper API + ElevenLabs/Cartesia или OpenAI TTS

## Где я сегодня ошибался — для будущего себя и для отца

За эту сессию я несколько раз перестраивал картину под давлением правильных уточнений отца. Это сигнал на будущее.

1. **Цеплялся за документ Family AI Tutor как за план**, хотя отец с самого начала сказал «это идея, реализация гибкая». Перестроил после нескольких ходов
2. **Дал устаревшую оценку статуса Матвея** — взял контекст из ранних чатов («шаги 1-3»), хотя на самом деле шаги 1-7 закрыты. На будущее: при возврате в проект через время — отец в первом сообщении кратко указывает текущий статус Матвея, это снимает риск работы с устаревшей памятью
3. **Преувеличил сложность Writing-модуля** — назвал «намного дороже» Speaking, хотя архитектурно это та же задача
4. **Механически наследовал «не делать видео»** из документа, не пересматривая под нашу рамку. Видео-модуль реализуем и ценен, отложен правильно — но по приоритету, не по запрету
5. **Несколько раз писал длинными ответами** там, где отец просил короче. Стиль работы отца: «меньше воды, конвертами, по делу»

Полезное о методе работы отца как продакта в этой сессии: **сомневается в чужих формулировках, даже если они звучат разумно**. Не принимает на веру ни документ, ни рекомендации Claude. Каждое решение проходит через личное переосмысление. Это правильное продакт-мышление.
